     ## Swift 6 concurrency (important)

     The project builds with **`SWIFT_STRICT_CONCURRENCY = complete`** and
     **`SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`**. Practical consequences for
     any new code:
     - Top-level declarations are implicitly `@MainActor` unless you opt out — 
     UI code "just works" on the main actor, but background/async work must be 
     explicitly moved off it (e.g. `nonisolated`, `Task.detached`, or 
     actor-isolated types).
     - Types crossing actor boundaries must be `Sendable`. Concurrency 
     violations are hard errors, not warnings — assume the compiler will reject 
     unsound code.

     The per-target configs in RepoExplorer.xcodeproj/project.pbxproj currently

     - SWIFT_VERSION = 5.0 on all three targets (lines 422, 454, 475, 497,

     Empirically-verified concurrency facts that drive the design

     Under SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor +
     SWIFT_APPROACHABLE_CONCURRENCY = YES + strict-complete:
     1. Sendable ≠ off-main. A Sendable struct conforming to a Sendable protocol
     still gets its methods inferred @MainActor (isolation inferred from the
     conformance) → JSON decode would run on the main thread. Sendability and
     actor-isolation are orthogonal.
     2. To run networking + decode genuinely off-main you need @concurrent, not
     just nonisolated. With approachable concurrency on, a bare nonisolated 
     async func runs on the caller's executor (= main when called from the VM).
     @concurrent nonisolated func on the client method pushes it to the
     concurrent pool (verified CONCURRENT_DECODE_ON_MAIN=false).
     3. Models decoded off-main must be nonisolated, else #IsolatedConformances
     error ("main actor-isolated conformance of X to Decodable cannot be used in
     nonisolated context").
     4. The history store must be a nonisolated actor (a plain actor's init is
     @MainActor-inferred and can't assign its isolated stored properties).
     5. Do not use .convertFromSnakeCase — it maps html_url→htmlUrl, so a
     property named htmlURL/avatarURL/spdxID throws keyNotFound at runtime. Use
     explicit snake_case CodingKeys.
     6. Debounce must be structured, run inline within the SwiftUI .task(id:)
     scope — an inner unstructured Task {} escapes .task cancellation, so
     leaving the screen mid-flight would not cancel the in-flight request.
     7. On CancellationError, derive a stable status (repos.isEmpty ? .idle : 
     .loaded); restoring a captured .loading strands the spinner after
     back-navigation.
     8. MockURLProtocol must be a nonisolated final class on a per-test
     ephemeral URLSession (its Foundation overrides are nonisolated; avoid
     shared static mutable state).

⏺ Good question — it comes down to which observation system the view model uses,
  and the project's "no Combine" constraint.

  The short answer

  @ObservedObject only works with ObservableObject, which is built on Combine.
  Our SearchViewModel uses the newer @Observable macro (the Observation
  framework), so @ObservedObject literally wouldn't apply to it — and using
  ObservableObject would violate your "no Combine" rule. @Bindable is the
  Observation-framework counterpart you reach for when a view needs two-way 
  bindings to an @Observable object it doesn't own.

  Why each piece is what it is

  SearchViewModel.swift:
  @MainActor
  @Observable                 // <- Observation framework, NOT ObservableObject
  final class SearchViewModel { var query = "" ... }

  @ObservedObject/@StateObject/@Published are the Combine-era trio
  (objectWillChange is a Combine publisher). Pairing them with an @Observable
  type won't compile, and pulling in ObservableObject would bring Combine back
  in — which you explicitly ruled out.

  Ownership: who needs which wrapper

  The App owns the instance; the view only reads/binds it:

  // RepoExplorerApp.swift — OWNER
  @State private var searchViewModel =
  AppDependencies.live.makeSearchViewModel()
  ...
  SearchView(viewModel: searchViewModel)

  // SearchView.swift — NON-OWNER, needs a binding
  @Bindable var viewModel: SearchViewModel
  .searchable(text: $viewModel.query)   // $ requires @Bindable

  The reason it's @Bindable and not just a plain let viewModel is the
  $viewModel.query binding that .searchable needs. For read-only observation, a
  plain let would already work (the @Observable macro makes SwiftUI track
  whatever properties body reads). @Bindable adds the $-projection so you can
  hand a Binding to the search field — without re-taking ownership (that's the
  App's @State).

  The mental map

  ┌──────────────────────┬──────────────────────────┬──────────────────────┐
  │         Role         │       Combine era        │  Observation (iOS    │
  │                      │                          │  17+, what we use)   │
  ├──────────────────────┼──────────────────────────┼──────────────────────┤
  │ Model declaration    │ class: ObservableObject  │ @Observable class    │
  │                      │ + @Published             │                      │
  ├──────────────────────┼──────────────────────────┼──────────────────────┤
  │ View owns it         │ @StateObject             │ @State               │
  ├──────────────────────┼──────────────────────────┼──────────────────────┤
  │ View observes (no    │ @ObservedObject          │ plain let            │
  │ binding)             │                          │                      │
  ├──────────────────────┼──────────────────────────┼──────────────────────┤
  │ View needs $         │                          │                      │
  │ bindings, doesn't    │ @ObservedObject          │ @Bindable ← our case │
  │ own                  │                          │                      │
  └──────────────────────┴──────────────────────────┴──────────────────────┘

  Bonus: beyond avoiding Combine, @Observable tracks reads per-property, so a
  view body only re-renders when a property it actually reads changes — more
  precise (and faster) than ObservableObject, which invalidates on any
  @Published change.

  So @Bindable here = "modern, Combine-free, give me $viewModel.query without
  claiming ownership."

❯ are we storing the api result with the search history?                        

⏺ No — we store only the query text and a timestamp, not the API results.

  The persisted unit is RecentSearch:

  // Models/RecentSearch.swift
  struct RecentSearch: Codable, Sendable, Identifiable, Hashable {
      var id: String { query }
      let query: String
      let date: Date
  }

  UserDefaultsSearchHistoryStore persists a JSON array of those — just [{query, 
  date}]. The [Repository] results from the search response are never written to
  disk or kept in the store.

  When a user taps a recent search, selectRecent(_:) sets the query and bumps
  the retry token, which re-triggers queryChanged() → a fresh network search for
  that query. So results are re-fetched live, not restored from a cache.

  This is intentional and matches the brief ("remember what they've searched
  for"):
  - Pros: recent searches always return up-to-date results (stars/new repos
  change), the stored data is tiny, and there are no stale-cache or
  model-migration concerns if Repository changes.
  - Cons: tapping a recent search needs a network round-trip (and counts against
  GitHub's unauthenticated ~10 searches/min), and there's no offline view of
  past results.

  If you'd instead want results cached (instant/offline re-display of a prior
  search), that's a deliberate design change — I'd cache the SearchResponse per
  query (with a freshness/expiry policy) and have selectRecent show the cached
  results immediately, optionally refreshing in the background. That's a bigger
  feature than Phase 3's "remember searches," so I left it out. Want me to add
  result caching?

          rwise: `#IsolatedConformances` error). Protocol requirements cr
          ossing into the VM are `nonisolated`.
